## Problem: TMS 2.0 Kundenliste - Sortierung & Suche funktionieren nicht

### Kontext:
- **App:** Next.js 16 + Supabase (self-hosted) + Docker
- **URL:** https://tms.gudel-werkzeuge.de/kunden
- **Feature:** PROJ-11 Kundendetailseite
- **Docker:** Container `tms` auf Port 3000, Traefik als Reverse Proxy

### Aktueller Stand:
1. ✅ Code-Änderungen sind auf dem Host in `/home/botti/projects/tms-2.0/src/lib/actions/partners.ts`
2. ✅ Docker Build war erfolgreich (mit `--no-cache`)
3. ✅ Container läuft (`docker ps | grep tms`)
4. ❌ Aber: **Die Seite zeigt immer noch falsche Sortierung**

### Root Cause (vermutet):
Docker kopiert beim `COPY . .` Schritt im Dockerfile die `.next/` Build-Artefakte vom Host. Wenn auf dem Host ein altes `.next/` Verzeichnis existiert, werden diese alten kompilierten Dateien in den Container kopiert, statt dass Next.js die neuen Dateien kompiliert.

**Wichtig:** `rm -rf .next` auf dem Host wurde bereits gemacht, aber das Problem besteht weiter.

### Dateien, die geändert wurden:
- `src/lib/actions/partners.ts` - Server Action für Kundenliste
- `src/app/kunden/page.tsx` - Kundenlisten-Seite
- Beide haben `export const dynamic = "force-dynamic"`

### Nächster Schritt:
Prüfe ob die Änderungen WIRKLICH im Container sind:
```bash
docker exec tms cat /app/src/lib/actions/partners.ts | grep "idsToLoad"
docker exec tms cat /app/.next/server/chunks/ssr/_*.js | grep "isNumeric"
```

Falls nicht → Das Problem ist, dass Docker den `COPY . .` Schritt cached. Lösung: `docker system prune -a` oder einen Cache-Bust einbauen.

Falls doch → Das Problem liegt woanders (z.B. Supabase Query, Daten).

---

**Wichtig:** Der Service Role Key für Supabase ist in `/home/botti/projects/tms-2.0/.env.local` hinterlegt.
